# Write 工具真实 LLM 速率流式 — 排查 + 修复计划

## 现状结论（已查明）

经过对全链路的代码 review，**应用层每一层都是逐事件转发 + flush 的**，没有发现任何人为 sleep 或 debounce：

| 层 | 文件:行 | 行为 |
|----|---------|------|
| Provider `read_sse` | `openai.rs:474` | 每个 arguments delta 立刻 `sender.send(ToolCallDelta)` ✅ |
| Provider mpsc | `openai.rs:406` | `cap 128` 有界，只 backpressure，不 batch ✅ |
| EventStream mpsc | `event_stream.rs:12` | `cap 256` ✅ |
| agent_loop 处理 | `agent_loop.rs:1327` | 逐事件 `on_tool_call_delta` 转发，无 buffer ✅ |
| Worker output | `ion_worker.rs:3459` | `writeln + flush`（每次 flush）✅ |
| Host reader | `worker_registry.rs:566` | `lines().next_line()` 逐行，立刻 `try_send` ✅ |
| Host socket 转发 | `ion.rs:3349-3350` | 每事件 `write_all + flush` ✅ |
| `cmd_subscribe` | `ion.rs:2026` | `println!(pretty_json)` — Rust 的 `LineWriter` 每行 flush ✅ |
| Python 代理 | `proxy_v2.py:118` | 每事件 `wfile.write + flush` ✅ |

**但仍观察到 97 chunks（直连 curl）→ 2-4 events（ion 链路）的损失。** 一定有某个点在丢/合并事件。

## 三种可疑原因（按可能性排序）

### 嫌疑 #1：`try_send` 在 256 channel 满时静默丢事件（最可能）
- `worker_registry.rs:593` 的 `let _ = sub.try_send(msg.clone());` — 当 subscriber 慢或 channel 满（256），事件直接被丢弃。
- **触发条件**：如果 host 主线程的 socket 转发 loop 被某处阻塞（比如全局 `sub_registry.lock().await` 在 worker_registry.rs:589 的竞争），多个 tool_call_delta 同时到达，channel 瞬间满，后面全部被 drop。

### 嫌疑 #2：reqwest Client 缺 `tcp_nodelay(true)` — 但只影响出站
- `openai.rs:251-253` `Client::builder().timeout(180s).build()` 没设 `tcp_nodelay`。
- 但 Nagle 算法只影响 **出站小包**，不影响入站流。对 SSE 入站流不是问题。**优先级低**，但仍应补上（影响出站 ACK 时机）。

### 嫌疑 #3：DeepSeek 经 opencode 代理可能合并 chunks
- 用户曾提到"直连 curl 得到 97 chunks"——但这个直连 curl 是对 DeepSeek 原生 API 还是对 opencode.ai？如果 ion 走 opencode 而 curl 直连 DeepSeek，差异可能就在代理层（不在 ion）。

## 实施计划（按顺序，先诊断后修复）

### 第 1 步：**加诊断计数器，找到真正的丢事件点**（不改逻辑，只加日志）

在 4 个关键节点加 stderr 日志（用 eprintln! 走 stderr，不污染 stdout JSONL）：

**文件**: `ion-provider/src/provider/openai.rs`
- `read_sse` 函数顶部：每次进 while 循环打印 chunk 大小（`[stream-debug] bytes_chunk N`）
- 每个 `sender.send(ToolCallDelta)` 前：`[stream-debug] emit tool_call_delta len={delta.len()}`

**文件**: `src/bin/ion_worker.rs`
- `on_tool_call_delta` 内：`eprintln!("[stream-debug] worker emit tool_call_delta name={name}")`

**文件**: `src/worker_registry.rs`
- reader task 转发前：`eprintln!("[stream-debug] host forward event type=...")`
- `try_send` 失败时打印：`eprintln!("[stream-debug] host DROP event (channel full)")`

**验证**：起 host → 发"写 30 行文件" → 看 host stderr 文件（`~/.ion/host.stderr` 或类似）里：
- 如果 provider 层只有 2-4 个 `[stream-debug] emit tool_call_delta` → 问题在 provider/reqwest（嫌疑 #2 或代理合并）
- 如果 provider 有 97 个但 host forward 只有 2-4 个 → worker stdout 传输或 host reader 丢
- 如果 host forward 有 97 个但 `try_send` 有 drop → channel 容量不够
- 如果都正常但 subscribe 看不到 → socket 转发或 Python 代理吞了

### 第 2 步：**根据诊断结果精准修复**

根据第 1 步定位的层次：

**场景 A — provider 层就只有 2-4 个**（嫌疑 #2/#3）：
- 给 reqwest 加 `.tcp_nodelay(true)`（5 个 provider 都加）—— 解决出站 ACK 延迟
- 如果仍不行，说明是 DeepSeek/opencode 代理合并，不是 ion 能控制的

**场景 B — try_send 在丢事件**（嫌疑 #1）：
- `worker_registry.rs:980, 994` 的 `mpsc::channel(256)` → `mpsc::channel(4096)`（tool_call_delta 流量大）
- `try_send` 失败时改为 `send().await`（背压而不丢）—— 但要小心不要让 reader task 永久阻塞

**场景 C — host forward 正常但 socket 转发慢**：
- 检查 `ion.rs:3349` 的 socket 转发 loop 是否有 lock 竞争
- 改 `write_half.write_all + flush` 为批量写（一次写多个，但单次 syscall）—— 但这会重新引入 batch，**不推荐**

### 第 3 步：**用真实数据复测**

```bash
# 启动 host
ion serve --provider opencode --model deepseek-v4-flash 2> /tmp/host.stderr &

# 创建 session + 发 prompt（另一终端）
ion rpc --method create_session
ion rpc --session <sid> --method prompt --params '{"text":"用 write 工具创建一个 30 行的文件 demo.txt，每行内容是 line N"}'

# subscribe 看事件
ion subscribe --session <sid> | grep tool_call_delta | wc -l

# 看 host stderr 的 stream-debug 计数
grep "stream-debug" /tmp/host.stderr | grep "emit tool_call_delta" | wc -l
grep "stream-debug" /tmp/host.stderr | grep "DROP" | wc -l
```

**预期健康指标**：
- `emit tool_call_delta` ≈ 30-50 次（DeepSeek 每 token 一次 args 增量）
- `DROP` = 0（没有丢事件）
- `subscribe | grep tool_call_delta` ≈ 30-50 次

## 不会做什么

- ❌ **不加人为 sleep** —— 用户明确要求 100% 真实 LLM 速率
- ❌ **不重写 Python 代理** —— 已确认它正确 flush
- ❌ **不重写 EventSource** —— 标准 API 已经正确
- ❌ **不动 agent_loop 的 select! abort 机制** —— 200ms 只在没事件时触发，不影响正常流

## 验收标准

1. `ion subscribe --session <sid> | grep tool_call_delta | wc -l` 应该 > 20（不是 2-4）
2. 前端"写 30 行文件"时，`+N` 数字应该在 1-2 秒内多次跳动（不是秒出 30）
3. 跳动速率跟 DeepSeek 实际生成速率一致
4. 不引入任何人为延迟

如果诊断后发现是 DeepSeek/opencode 上游合并 chunks（ion 无法控制），则如实报告"这是上游 API 的限制"，并讨论替代方案（比如让前端在收到第一个 tool_call_delta 后用 CSS 动画显示"working..."指示器，但不在数字上造假）。
